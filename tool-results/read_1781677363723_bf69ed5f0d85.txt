     7→
     8→This report maps 76 skills across 11 categories to 8 distinct user types, providing each with a complete stack profile including: recommended skills (Core / Extended / Power-User), agent skill tier assignment and progression, primary agent mode, MCP server stack, learning path, token budget, common pitfalls, and an optimal 1-hour workflow. The analysis draws from existing persona research, the 4-skill manifest architecture (SKILL_01–04), and 9 pre-built MCP stacks.

    11→- `persistent-memory` is the #1 most-universal skill (recommended for 8/8 user types)
    12→- `superpowers` and `context-compressor` are #2 most-universal (7/8 user types each)
    13→- Backend/API Engineer, Full-Stack Generalist, and Data Scientist are **new user types** not covered in prior persona research, requiring original stack design
    14→- The 4 Agent Skill Tiers map cleanly to user-type progression paths: all users start at SKILL_01, then diverge based on their primary workflow
    15→- Token budgets range from 4,400 (SKILL_01 only) to 12,500 (all 4 skills loaded) — most users need 2 active skills at any time (~10,000 tokens)

    21→### Profile Description
    22→The Solo Indie Hacker is a non-technical or semi-technical founder building their first product. They typically have a business or marketing background with enough coding knowledge to be dangerous — they can modify existing code, follow tutorials, and use AI-assisted development, but they lack the deep architectural knowledge to make sound technical decisions independently. They operate under extreme resource constraints: bootstrapped budgets, no team, and a "ship or die" timeline. Mobile-first usage is their reality — they think about their product during commutes, sketch features on their phone, and need the AI to be their entire technical team. They need guided, opinionated workflows that make decisions for them when possible and present clear choices when necessary. Their psychological profile is scrappy, impatient with abstraction, and deeply motivated by seeing tangible progress.

   109→### Common Pitfalls
   110→1. **Over-engineering the MVP** — The indie hacker's biggest risk is adding features instead of shipping. The `to-prd` skill should enforce a strict "3 features max" rule for v1. Without this, they'll build a Swiss-army knife nobody wants to use.
   111→2. **Skipping the humanizer** — AI-generated marketing copy sounds like AI-generated marketing copy. Users can tell. The `humanizer` skill is not optional — it's the difference between "clearly written by a robot" and "sounds like a real person who understands my problem."
   112→3. **Ignoring persistent-memory** — Re-explaining your entire codebase every session is the #1 time-waster for indie hackers. Load `persistent-memory` from Day 1, not Week 3.
   113→4. **Loading too many skills** — Analysis paralysis. 7 core skills are enough to ship. Don't add extended skills until the core loop is working.
   114→5. **Building before validating** — Using `superpowers` to build features nobody asked for. The correct order is: to-prd → jtbd-research → superpowers. Not the reverse.

   284→## USER TYPE 3: BACKEND/API ENGINEER
   285→
   286→### Profile Description
   287→The Backend/API Engineer is a systems thinker who builds APIs, databases, microservices, and the infrastructure that powers applications. They are deeply technical — comfortable with distributed systems, database design, API design patterns (REST, GraphQL, gRPC), message queues, caching strategies, and performance optimization. They think in terms of latency percentiles, throughput, error rates, and reliability. Their daily work involves designing schemas, writing service endpoints, implementing authentication/authorization, optimizing queries, debugging production issues, and writing integration tests. They are the least visually-oriented user type — they care about correctness, performance, and reliability, not pixel perfection. Their biggest challenges are: (1) managing API versioning and backward compatibility, (2) debugging distributed system failures across service boundaries, (3) maintaining database performance under load, and (4) ensuring security across the entire API surface.
